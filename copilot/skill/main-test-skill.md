# Main Test Skill — Hub Frontend (QA + Test Prompting)

Summary
- This skill helps craft deterministic test plans, generate unit/integration/E2E tests, produce QA reports, and produce clear bug reproduction steps for this repository.

Repository context (use in prompts)
- Framework: Next.js (app router), TypeScript, Tailwind CSS.
- Key locations to reference in prompts:
  - src/app — routes and server/client pages
  - src/components/NavBar.tsx — navigation component
  - src/app/layout.tsx and src/app/globals.css — layout and styles
  - src/lib/api.ts and src/lib/utils.ts — API helpers and utilities
  - src/store/authStore.ts — auth & global store patterns
  - src/types/index.ts — shared TypeScript interfaces

Output contract (required before closing a task)

1. Tests and changes are delivered as unified diffs/patches only.
2. New shared types go in src/types/index.ts.
3. Tests are deterministic and fast: external services mocked, no sleeps, isolated state.
4. Any new dev-dependencies must be proposed and approved before installation.

How to use

1. Ask for a 3–6 step test plan for the target file or feature.
2. Confirm test frameworks to use (existing project choice or recommend `vitest` for unit tests, `playwright` for E2E).
3. Request patches only after approving the plan.

Persona (system prompt example)
You are a senior QA engineer and test-writer for this repo. Produce deterministic, fast tests that follow project conventions. State the test runner and test style. Provide a short plan, then patches for tests and any required small helper modules or fixtures.

Workflows & templates

- Unit test for a component
  - Prompt: "Write unit tests for ComponentX at src/components/ComponentX.tsx. Use React Testing Library, assert props, events, and accessibility. Return a plan and patches."

- Integration test for API helper
  - Prompt: "Create integration tests for src/lib/api.ts covering success, 4xx/5xx responses, and malformed bodies. Mock fetch and assert error handling. Return patches and fixtures."

- E2E scenario
  - Prompt: "Write an E2E test that uploads a document via the /documents page, verifies a preview, and confirms the item appears in the list. Use Playwright (or the project's E2E tool). Mock backend upload responses where possible."

- Bug reproduction + fix flow
  - Prompt: "Investigate failing behavior in src/app/chat/[sessionId]/page.tsx. Provide reproduction steps, a failing test, and a minimal fix with patches."

Testing conventions

- Ask for CI and runner expectations before adding dev-deps.
- Preferred test locations: alongside files (`Component.test.tsx`) or a `__tests__` folder.
- Fixtures and factories: put under tests/fixtures or src/tests/fixtures and export types to src/types/index.ts.
- Keep unit tests fast and isolated; slow/integration tests should be labeled and run in a separate CI job.

Quality checklist to require from the assistant

- Provide a 3–6 step plan before coding.
- Return unified diffs/patches only.
- Include deterministic test code (no sleeps, no network calls without mocks).
- Add minimal helper mocks/fixtures and type them in src/types/index.ts.
- Provide reproduction steps and severity for any bugs found.

Examples (copy-paste prompts)

- "Plan and implement unit tests for src/components/NavBar.tsx. Include tests for logged-in vs logged-out states and navigation links. Return a plan and patches."
- "Create integration tests for src/lib/api.ts that simulate a 500 response and assert retry logic."
- "Write a Playwright E2E that covers document upload on /documents page. Mock the backend upload response and assert UI flow."

Bug report template

**Title:** [Component] Brief description
**Severity:** Critical | High | Medium | Low
**Steps to Reproduce:** 1. ... 2. ...
**Expected:** ...
**Actual:** ...
**Environment:** OS/browser/version
**Evidence:** logs/screenshot/test output
